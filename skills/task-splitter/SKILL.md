# Task Splitter Skill

大きな要件・機能をエージェントが実行しやすい Atomic Task に分解するスキル。

## Atomic Task の定義

- **10〜15分** で完結する
- **単一の責任** を持つ（コード変更 + テスト + ドキュメントを1つのタスクに混在させない）
- **独立してテスト可能** （完了したかどうかを判定できる）
- **依存関係が明確** （先にやるべきタスクが何かが分かる）

## ワークフロー

### Step 1: 要件を理解する

ユーザーの要求を読み、以下を明確にする:

- **ゴール**: 何が完成したら「done」か
- **スコープ**: 何を含み、何を含まないか
- **制約**: 技術スタック、期限、依存関係
- **不明点**: 判断が必要なものはここで確認する

### Step 2: 分解する

以下の手順でタスクを生成する:

1. **バックエンド → フロントエンド → テスト → ドキュメント** の順で考える
2. **データモデル → API → ビジネスロジック → UI** の依存順を意識する
3. 各タスクが 15 分を超えそうなら、さらに分割する
4. 共通ユーティリティ・共通コンポーネントは先行タスクとして切り出す

### Step 3: タスクリストを出力する

```markdown
## タスクリスト: [機能名]

推定合計: X タスク / 約 Y 時間

### Phase 1: [フェーズ名]
- [ ] **[T1]** [タスク名] — [完了条件]
- [ ] **[T2]** [タスク名] — [完了条件] (depends: T1)

### Phase 2: [フェーズ名]
- [ ] **[T3]** [タスク名] — [完了条件]
```

### Step 4: 優先順位を提案する

- **クリティカルパス** (他のタスクをブロックするもの) を特定して強調する
- **リスクの高いタスク** (不明点が多い・初めて触る領域) は早期に着手を推奨する
- **独立タスク** (並列実行可能なもの) を明示する

## Linear Issue への変換（任意）

ユーザーが「Linear に起票して」と言った場合、各タスクを Linear issue のフォーマットに変換する:

```
タイトル: [T1] [タスク名]
説明: [完了条件・背景]
優先度: [Urgent/High/Medium/Low]
ラベル: [feature/fix/docs]
```

## タスク粒度の目安

| タスク例 | 適切 | 理由 |
|---------|------|------|
| `UserモデルにemailフィールドをMigrationで追加` | ✅ | 10分 |
| `認証機能全体を実装` | ❌ | 大きすぎる（複数日） |
| `ログイン API エンドポイントを実装` | ✅ | 15分 |
| `エラーハンドリングを整理` | ⚠️ | スコープが曖昧 → 具体化する |

## Gotchas

- タスクを細かくしすぎない（2-3分のタスクは管理コストが見合わない）
- 「調査タスク」は最大 30 分で時間ボックスを設ける
- リファクタリングは機能追加タスクと混ぜない（別タスクにする）
- テストは実装タスクと 1:1 で対応するタスクを立てる
